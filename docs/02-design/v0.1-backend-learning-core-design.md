# Detailed Design: v0.1 Backend Learning Core

**Source**: `architecture-to-design` skill native output

## Overview

v0.1 Backend Learning Core 是 Feynman Learning Studio 的第一个 backend implementation slice。它提供 Learning Topic 和 Learning Experiment 的 REST API、统一 response/error model、MyBatis persistence、H2 local tests 和 Swagger/OpenAPI inspection。

## Source Architecture

- Architecture: `docs/02-design/v0.1-backend-learning-core-architecture.md`
- Architecture style: layered modular monolith。
- Runtime stack: Spring Boot 3.x, Maven, MyBatis, Druid, H2, OpenAPI/Swagger。

## Design Assumptions

- 当前没有已有 backend code，本设计面向新建 `backend/`。
- v0.1 不实现 frontend 或 auth。
- v0.1 使用 physical delete。
- 被 LearningExperiment 引用的 LearningTopic 不允许删除。
- 默认 pageNo 为 1，pageSize 为 10，pageSize 上限为 100。
- List 默认排序为 `createdAt DESC, id DESC`。

## Design Scope

范围内：

- Spring Boot backend scaffold。
- Common API foundation。
- Learning Topic module。
- Learning Experiment module。
- MyBatis persistence。
- H2 test profile。
- Swagger/OpenAPI。

范围外：

- Frontend。
- Auth。
- User/team/member model。
- External AI provider。
- Production deployment。

## Module Design

### Common API Foundation

职责：

- 定义 `ApiResponse<T>`。
- 定义 `PageResponse<T>`。
- 定义 error codes。
- 定义 business exception。
- 定义 global exception handler。
- 将 validation、business、database、unexpected errors 转成统一 envelope。

### Learning Topic Module

职责：

- 创建 topic。
- 按 id 查询 topic。
- 分页查询 topic。
- 更新 topic。
- 删除 topic。
- 校验 slug uniqueness。
- 拒绝删除被 experiment 引用的 topic。

### Learning Experiment Module

职责：

- 创建 experiment。
- 按 id 查询 experiment。
- 分页查询 experiment。
- 按 topicId filter experiments。
- 更新 experiment。
- 删除 experiment。
- 更新 experiment status。
- 校验 topic reference。
- 校验 difficulty 和 status transition。

### Persistence Module

职责：

- 使用 MyBatis mapper 执行 parameterized SQL。
- 提供 schema for H2 tests。
- 支持 topic 和 experiment CRUD。
- 支持 pagination count/list。

### OpenAPI Module

职责：

- 暴露 Swagger UI。
- 暴露 OpenAPI JSON。
- 展示 endpoint、request、response 和 error format。

## API / Interface Design

### Topic API

| Operation | Method | Path |
|---|---|---|
| Create Topic | POST | `/api/topics` |
| Get Topic | GET | `/api/topics/{id}` |
| List Topics | GET | `/api/topics` |
| Update Topic | PUT | `/api/topics/{id}` |
| Delete Topic | DELETE | `/api/topics/{id}` |

### Experiment API

| Operation | Method | Path |
|---|---|---|
| Create Experiment | POST | `/api/experiments` |
| Get Experiment | GET | `/api/experiments/{id}` |
| List Experiments | GET | `/api/experiments` |
| Update Experiment | PUT | `/api/experiments/{id}` |
| Update Status | PATCH | `/api/experiments/{id}/status` |
| Delete Experiment | DELETE | `/api/experiments/{id}` |

## Data Design

- `LearningTopic` 1:N `LearningExperiment`。
- `LearningTopic.slug` is unique。
- `LearningExperiment.topicId` references `LearningTopic.id`。
- `LearningExperiment.status` follows the v0.1 transition map。

## Workflow / Execution Design

### Create Topic

1. Controller receives request。
2. Validation checks request shape。
3. Service checks slug uniqueness。
4. Mapper inserts topic。
5. API returns success envelope。

### Delete Topic

1. Controller receives topic id。
2. Service checks topic exists。
3. Service checks experiment reference count。
4. If referenced, return `TOPIC_IN_USE`。
5. If not referenced, mapper physically deletes topic。

### Create Experiment

1. Controller receives request。
2. Validation checks request shape。
3. Service checks topic exists。
4. Service validates difficulty/status。
5. Mapper inserts experiment。
6. API returns success envelope。

### Update Experiment Status

1. Service loads current experiment。
2. Service validates target status。
3. Service checks transition map。
4. Mapper updates status。
5. API returns updated experiment。

## Integration Design

- **H2**: test profile database。
- **Druid**: datasource baseline。
- **Swagger/OpenAPI**: local API inspection。
- **PostgreSQL**: future integration database direction; not implemented in v0.1。

## Security / Audit / Reliability Design

- No auth in v0.1。
- Validate all request bodies, path variables, and query parameters。
- Use parameterized SQL only。
- Do not expose stack traces, SQL, class names, connection strings, or local paths in API errors。
- Log unexpected exceptions once at boundary。

## Validation and Error Handling

| Error Code | Trigger |
|---|---|
| `VALIDATION_ERROR` | request validation failure |
| `TOPIC_NOT_FOUND` | topic id not found |
| `TOPIC_IN_USE` | topic is referenced by experiments |
| `EXPERIMENT_NOT_FOUND` | experiment id not found |
| `DUPLICATE_TOPIC_SLUG` | topic slug already exists |
| `INVALID_TOPIC_REFERENCE` | experiment references non-existing topic |
| `INVALID_EXPERIMENT_STATUS` | target status is unsupported |
| `INVALID_STATUS_TRANSITION` | transition is unsupported |
| `DATABASE_ERROR` | database access failure |
| `SYSTEM_ERROR` | unexpected error |

## Testing Considerations

- Topic API integration tests。
- Experiment API integration tests。
- Validation and error-path tests。
- Status transition tests。
- H2-backed tests without Docker。

Required command:

```bash
cd backend
mvn test -Dspring.profiles.active=test
```

## Risks / Design Tradeoffs

- v0.1 physical delete is simple, but future learning history may require archive/soft delete。
- No auth limits v0.1 to local learning。
- H2 and PostgreSQL behavior may differ。
