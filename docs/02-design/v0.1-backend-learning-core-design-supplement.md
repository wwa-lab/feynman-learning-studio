# Project Supplement: v0.1 Backend Learning Core Design

**Supplements**: `docs/02-design/v0.1-backend-learning-core-design.md`

## 1. Repository-Specific Package Rule

Backend implementation must follow package-by-feature:

```text
backend/src/main/java/com/feynman/learningstudio/
  common/
    api/
    exception/
    validation/
  topic/
    controller/
    service/
    mapper/
    domain/
    dto/
  experiment/
    controller/
    service/
    mapper/
    domain/
    dto/
```

## 2. Implementation Handoff Notes

- Controller owns API request validation and delegates to service。
- Service owns business rules and transaction boundary。
- Mapper owns SQL only。
- DTO names should show direction: `CreateTopicRequest`, `TopicResponse`, etc。
- Mapper XML namespace must match mapper interface fully qualified name。

## 3. Verification Gate Additions

Required evidence:

- `mvn test -Dspring.profiles.active=test` output。
- local run output。
- Swagger UI reachable at `http://localhost:8080/swagger-ui/index.html`。
- OpenAPI JSON reachable at `http://localhost:8080/v3/api-docs`。
- Topic success API example。
- Experiment success API example。
- Validation or not found error example。

## 4. Documentation Gate Additions

Implementation must update:

- `docs/04-runbooks/v0.1-backend-learning-core-runbook.md`
- Feynman reflection notes
- draw.io diagram if implementation changes boundaries or relationships

## 5. Review Gate Additions

Before PR or completion:

- Codex reviews git diff。
- Security review checks validation, SQL safety, secret handling, and error disclosure。
- Claude Code independent review is expected for non-trivial backend implementation。
