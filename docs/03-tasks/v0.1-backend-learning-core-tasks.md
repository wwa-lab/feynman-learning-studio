# Implementation Task Breakdown

**Source**: `design-to-tasks` skill native output  
**Source Design**: `docs/02-design/v0.1-backend-learning-core-design.md`  
**System Name**: v0.1 Backend Learning Core

## Overview

Implement the first runnable backend slice for Feynman Learning Studio. The delivery objective is a Spring Boot 3.x backend with Learning Topic CRUD, Learning Experiment CRUD, status transition, unified API envelope, MyBatis persistence, H2 tests, and Swagger/OpenAPI inspection。

Planning assumptions:

- No existing backend code is present。
- v0.1 uses physical delete。
- Referenced topics cannot be deleted。
- No frontend or auth in v0.1。

## Source Design

The design defines a layered modular monolith with common API foundation, topic domain, experiment domain, persistence, and OpenAPI support. It targets local verification through H2-backed tests and local run evidence。

## Workstreams

- Backend foundation。
- Common API and error handling。
- Persistence and schema。
- Topic domain。
- Experiment domain。
- Testing。
- OpenAPI and documentation。

Recommended sequencing:

```text
foundation -> common API -> persistence -> topic -> experiment -> OpenAPI -> verification
```

## Task Details

### TASK-001: Create backend project skeleton

- **Objective**: Create a Spring Boot 3.x Maven backend project。
- **Scope**: `backend/`, application entry point, base package, Maven configuration。
- **Dependencies**: None。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Use JDK 17。

### TASK-002: Configure runtime and test dependencies

- **Objective**: Add dependencies required by the v0.1 design。
- **Scope**: Spring Web, Validation, MyBatis, Druid, H2, JUnit 5, OpenAPI/Swagger。
- **Dependencies**: TASK-001。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: No Docker-only dependencies。

### TASK-003: Implement common API envelope and exception handling

- **Objective**: Provide shared API response and error behavior。
- **Scope**: API response, pagination response, error codes, business exceptions, global exception handler。
- **Dependencies**: TASK-002。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Error responses must not expose internal details。

### TASK-004: Create H2 schema

- **Objective**: Provide database schema for local tests。
- **Scope**: `learning_topic`, `learning_experiment`, constraints, indexes。
- **Dependencies**: TASK-002。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Keep future PostgreSQL compatibility in mind。

### TASK-005: Implement MyBatis mapper layer

- **Objective**: Provide parameterized SQL persistence。
- **Scope**: CRUD, pagination count/list, topic existence check, experiment count by topic。
- **Dependencies**: TASK-004。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Avoid `SELECT *` and user input concatenation。

### TASK-006: Write Topic API integration tests

- **Objective**: Define Topic API behavior before implementation。
- **Scope**: create, read, list, update, delete, duplicate slug, invalid slug, not found, topic in use。
- **Dependencies**: TASK-003, TASK-004。
- **Owner type**: QA。
- **Priority**: Must。
- **Notes**: Follow TDD; tests should fail before TASK-007。

### TASK-007: Implement Topic API

- **Objective**: Make Topic tests pass。
- **Scope**: DTOs, domain model, service, controller, mapper integration。
- **Dependencies**: TASK-005, TASK-006。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Referenced topic delete returns `TOPIC_IN_USE`。

### TASK-008: Review Topic slice

- **Objective**: Check Topic implementation before adding Experiment complexity。
- **Scope**: correctness, validation, SQL safety, error response shape。
- **Dependencies**: TASK-007。
- **Owner type**: security。
- **Priority**: Should。
- **Notes**: Lightweight review checkpoint。

### TASK-009: Write Experiment API integration tests

- **Objective**: Define Experiment CRUD behavior。
- **Scope**: create, read, list, update, delete, invalid topic reference, validation failure, not found。
- **Dependencies**: TASK-007。
- **Owner type**: QA。
- **Priority**: Must。
- **Notes**: Follow TDD。

### TASK-010: Implement Experiment API

- **Objective**: Make Experiment CRUD tests pass。
- **Scope**: DTOs, domain model, service, controller, mapper integration。
- **Dependencies**: TASK-005, TASK-009。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Validate topic reference in service layer。

### TASK-011: Implement Experiment status endpoint

- **Objective**: Support status update API。
- **Scope**: `PATCH /api/experiments/{id}/status`, status validation, transition rule。
- **Dependencies**: TASK-010。
- **Owner type**: backend。
- **Priority**: Must。
- **Notes**: Unsupported transition returns `INVALID_STATUS_TRANSITION`。

### TASK-012: Write status transition tests

- **Objective**: Prove lifecycle behavior。
- **Scope**: allowed transitions, invalid status, invalid transition。
- **Dependencies**: TASK-011。
- **Owner type**: QA。
- **Priority**: Must。
- **Notes**: Include at least one unsupported transition。

### TASK-013: Configure Swagger/OpenAPI

- **Objective**: Make API locally inspectable。
- **Scope**: Swagger UI, OpenAPI JSON, endpoint metadata。
- **Dependencies**: TASK-007, TASK-010, TASK-011。
- **Owner type**: backend。
- **Priority**: Should。
- **Notes**: Use SpringDoc default paths。

### TASK-014: Capture local verification evidence

- **Objective**: Prove the slice works locally。
- **Scope**: test command output, local run, Swagger URL, Topic API example, Experiment API example, error example。
- **Dependencies**: TASK-013。
- **Owner type**: QA。
- **Priority**: Must。
- **Notes**: Evidence belongs in runbook or task notes, not generated logs。

- No auth means v0.1 is local-learning only。
- Physical delete may need redesign when learning history becomes a product object。
