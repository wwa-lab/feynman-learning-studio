**状态**: Draft

## 1. 目标

构建 Feynman Learning Studio 的第一个可运行 backend 切片。

这个切片要让工程师能够通过 Spring Boot API 管理 Learning Topic 和 Learning Experiment，通过 H2 本地测试验证行为，在本地启动服务，通过 Swagger 查看 API，并最终用 Feynman learning loop 解释实现过程。

## 2. 既有工程经验

这个实验把传统 Java CRUD 和 backend API 开发经验，转译成 Spec Driven AI Development 的工作流。

相关既有经验包括：

- 传统 Java CRUD service 设计。
- Controller、service、mapper、database 分层。
- SQL 表设计和本地数据库测试。
- API validation 和 exception handling。
- 本地运行与测试证据收集。
- 在验证行为之后解释实现选择。

## 3. 问题陈述

项目当前已经有规则和文档结构，但还没有可运行的产品切片。

第一次实现必须证明：这个项目可以从 spec 走到本地 backend 证据，而不是停留在文档或变成泛泛的 CRUD demo。Learning Topic 和 Learning Experiment 是启动核心循环所需的最小产品对象：

```text
topic -> experiment -> spec/implementation work -> local test/run evidence -> Feynman reflection
```

## 4. 范围

范围内：

- `backend/` 下的 Spring Boot 3.x backend 项目。
- JDK 17 和 Maven。
- MyBatis persistence，使用 H2 支持本地测试。
- Druid datasource integration，作为 backend baseline 的一部分。
- Flyway database migration，用于统一管理 schema。
- Learning Topic CRUD。
- Learning Experiment CRUD。
- Learning Experiment status update。
- 统一 API response envelope。
- 统一 exception handling。
- API 边界使用 Jakarta Bean Validation。
- OpenAPI/Swagger documentation。
- 本地 backend runbook。

范围外：

- Frontend implementation。
- Authentication、authorization、users、members、teams、roles。
- TransformationPath、AgentWorkflow、RunEvidence、LocalRun、LocalTest、Reflection 作为独立持久化 domain object。
- Redis、message queue、Nacos、gateway、distributed services、Docker-only dependencies。
- 外部 AI provider integration。
- Production deployment。
- PostgreSQL integration，仅记录后续方向。

## 5. User / Engineer Stories

### Story 1: 管理 Learning Topics (Priority: P1)

作为工程师，我希望创建、查看、列表查询、更新和删除 Learning Topic，以便把 Learning Experiment 组织到稳定的 transformation area 下。

**独立测试**: H2-backed API integration tests 覆盖 topic create、read、list、update、delete、duplicate slug、validation failure 和 not found 行为。

**验收场景**:

1. Given 一个合法 topic request，when 工程师创建 topic，then API 返回包含已创建 topic 的成功 envelope。
2. Given 一个已存在 topic，when 工程师按 id 查询，then API 返回该 topic。
3. Given 多个 topic，when 工程师分页查询 topic，then API 返回确定顺序的 paginated result。
4. Given 一个已存在 topic，when 工程师更新其 name、slug、description 或 active flag，then API 返回更新后的 topic。
5. Given 一个已存在 topic 且 v0.1 没有阻塞 experiment 的规则，when 工程师删除它，then 后续读取返回 not found error。
6. Given duplicate slug、invalid slug、blank name 或 invalid id，when 提交请求，then API 返回用户友好的 error envelope。

### Story 2: 管理 Learning Experiments (Priority: P1)

作为工程师，我希望创建、查看、列表查询、更新、删除和变更 Learning Experiment 状态，以便每个可运行学习活动都有具体的 backend 记录。

**独立测试**: H2-backed API integration tests 覆盖 experiment create、read、list、update、delete、status update、invalid topic reference、validation failure、invalid status transition 和 not found 行为。

**验收场景**:

1. Given 一个已存在 topic 和合法 experiment request，when 工程师创建 experiment，then API 返回包含已创建 experiment 的成功 envelope。
2. Given invalid topic id，when 工程师创建 experiment，then API 返回稳定的 invalid topic reference error。
3. Given 已存在 experiments，when 工程师按 page 和可选 topic id 查询，then API 返回确定顺序的 paginated result。
4. Given 一个已存在 experiment，when 工程师更新 title、summary、topic、difficulty 或 related spec path，then API 返回更新后的 experiment。
5. Given 一个已存在 experiment，when 工程师通过允许的 transition 变更 status，then API 返回更新后的 status。
6. Given invalid status 或 invalid transition，when 提交请求，then API 返回用户友好的 error envelope。

### Story 3: 验证并解释 Backend Slice (Priority: P1)

作为正在实践 Spec Driven AI Development 的工程师，我希望获得本地 test 和 run evidence，以便用 Feynman loop 解释自己构建了什么。

**独立测试**: backend test command 本地通过，service 可以本地启动，Swagger 可访问，runbook 记录命令。

**验收场景**:

1. Given backend project，when 在 `backend/` 下运行 `mvn test -Dspring.profiles.active=test`，then H2-backed test suite 通过。
2. Given backend project，when 本地 service 启动，then Swagger/OpenAPI 可访问。
3. Given 一次完成的 local run，when 工程师阅读 runbook，then 命令和预期证据可复现。

## 6. 功能需求

- **FR-001**: 系统 MUST 在 `/api/topics` 下暴露 Learning Topic CRUD endpoints。
- **FR-002**: 系统 MUST 在 `/api/experiments` 下暴露 Learning Experiment CRUD endpoints。
- **FR-003**: 系统 MUST 在 `/api/experiments/{id}/status` 下暴露 Learning Experiment status update endpoint。
- **FR-004**: 系统 MUST 使用包含 `success`、`data`、`errorCode`、`errorMessage` 的 common envelope 返回所有 API response。
- **FR-005**: 系统 MUST 使用包含 `records`、`pageNo`、`pageSize`、`total` 的结构返回 paginated list response。
- **FR-006**: 系统 MUST 在 API boundary 验证 request body、path variable 和 query parameter。
- **FR-007**: 系统 MUST 验证 topic slug format，只允许 lowercase letters、numbers 和 hyphens。
- **FR-008**: 系统 MUST 防止 duplicate topic slug。
- **FR-009**: 系统 MUST 验证每个 experiment 引用已存在的 topic。
- **FR-010**: 系统 MUST 验证 experiment status value 和 status transition。
- **FR-011**: 系统 MUST 使用 MyBatis parameterized SQL，MUST NOT 拼接 user input 到 SQL。
- **FR-012**: 系统 MUST 按 feature package 分离 controller、service、mapper、domain 和 DTO responsibilities。
- **FR-013**: 系统 MUST 提供不依赖 Docker 或外部数据库的 H2-backed tests。
- **FR-014**: 系统 MUST 为已实现 endpoints 暴露 OpenAPI/Swagger documentation。
- **FR-015**: 系统 MUST 提供包含 setup、test、run、Swagger 和 troubleshooting notes 的 local runbook。
- **FR-016**: 系统 MUST 使用 Flyway 管理 database schema，MUST NOT 使用临时 `schema.sql` 管理 application schema。

## 7. Data / Domain Model

- **LearningTopic**: 稳定的 learning area 或 transformation topic。
  - `id`: numeric identifier。
  - `name`: human-readable topic name。
  - `slug`: unique lowercase URL/API-friendly key。
  - `description`: optional explanation。
  - `active`: topic 是否可用于创建 new experiments。
  - `createdAt`: creation timestamp。
  - `updatedAt`: last update timestamp。

- **LearningExperiment**: 挂在某个 topic 下的 runnable learning activity。
  - `id`: numeric identifier。
  - `topicId`: 对 `LearningTopic` 的 reference。
  - `title`: experiment title。
  - `summary`: short experiment description。
  - `difficulty`: learning difficulty，例如 `BEGINNER`、`INTERMEDIATE`、`ADVANCED`。
  - `status`: lifecycle status。
  - `specPath`: 可选的 active spec repository path。
  - `createdAt`: creation timestamp。
  - `updatedAt`: last update timestamp。

初始 experiment statuses：

```text
DRAFT -> READY -> RUNNING -> COMPLETED
DRAFT -> ARCHIVED
READY -> ARCHIVED
RUNNING -> ARCHIVED
```

v0.1 故意保持 status transition 很小。Reopening、review workflow、evidence status 和 reflection status 暂不进入 v0.1。

## 8. API / Interface Contract

Topic endpoints:

```text
POST   /api/topics
GET    /api/topics/{id}
GET    /api/topics?pageNo=1&pageSize=10
PUT    /api/topics/{id}
DELETE /api/topics/{id}
```

Experiment endpoints:

```text
POST   /api/experiments
GET    /api/experiments/{id}
GET    /api/experiments?pageNo=1&pageSize=10&topicId={topicId}
PUT    /api/experiments/{id}
PATCH  /api/experiments/{id}/status
DELETE /api/experiments/{id}
```

Successful response shape:

```json
{
  "success": true,
  "data": {},
  "errorCode": null,
  "errorMessage": null
}
```

Paginated response shape:

```json
{
  "success": true,
  "data": {
    "records": [],
    "pageNo": 1,
    "pageSize": 10,
    "total": 0
  },
  "errorCode": null,
  "errorMessage": null
}
```

Error response shape:

```json
{
  "success": false,
  "data": null,
  "errorCode": "VALIDATION_ERROR",
  "errorMessage": "Request validation failed."
}
```

Required error codes:

```text
VALIDATION_ERROR
TOPIC_NOT_FOUND
TOPIC_IN_USE
EXPERIMENT_NOT_FOUND
DUPLICATE_TOPIC_SLUG
INVALID_TOPIC_REFERENCE
INVALID_EXPERIMENT_STATUS
INVALID_STATUS_TRANSITION
DATABASE_ERROR
SYSTEM_ERROR
```

## 10. Validation And Error Handling

- Topic `name` 必填，且必须有长度上限。
- Topic `slug` 必填、唯一，且必须匹配 lowercase letters、numbers 和 hyphens。
- Topic `description` 可选，但必须有长度上限。
- Pagination parameters 必须为正数且有上限。
- Experiment `topicId` 必填，且必须引用已存在 topic。
- Experiment `title` 必填，且必须有长度上限。
- Experiment `summary` 可选，但必须有长度上限。
- Experiment `difficulty` 必须是 supported values 之一。
- Experiment `status` 必须是 supported values 之一。
- Status update 必须遵守 allowed transition map。
- Not found error 不得暴露 stack trace、SQL、class name 或 local path。
- Database error 必须在 API boundary 转换成稳定、用户友好的 error envelope。
- Unexpected error 必须在 boundary 带诊断上下文记录一次日志，并返回 `SYSTEM_ERROR`。

## 14. Decisions

- **DEC-001**: 当 `LearningTopic` 已被 `LearningExperiment` 引用时，`DELETE /api/topics/{id}` MUST 拒绝删除，并返回 `TOPIC_IN_USE`。
- **DEC-002**: v0.1 使用 physical delete。`LearningExperiment` delete 直接删除；`LearningTopic` 未被引用时直接删除。
- **DEC-003**: Swagger UI 使用 SpringDoc 默认路径：`http://localhost:8080/swagger-ui/index.html`。
- **DEC-004**: OpenAPI JSON 使用 SpringDoc 默认路径：`http://localhost:8080/v3/api-docs`。
- **DEC-005**: H2 之后默认 integration database 方向是 PostgreSQL；v0.1 仍只实现 H2 local tests。
